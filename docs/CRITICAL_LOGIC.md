# 🌿 GreenLink v2 - CRITICAL LOGIC & SSOT (Single Source of Truth)

> **경고 (WARNING)**: 이 문서는 GreenLink 프로젝트의 **유일한 진실의 원천(SSOT)** 입니다. 모든 아키텍처 결정, 비즈니스 로직 수정, API 및 DB 스키마 설계는 이 문서의 원칙을 위배해서는 안 됩니다. 기존 로직을 수정해야 할 경우, 반드시 이 문서를 선제적으로 업데이트하고 팀(또는 사용자)의 승인을 받아야 합니다.

---

## 1. 아키텍처 및 성능 원칙 (Architecture & Performance)

본 프로젝트의 최우선 과제는 **극단적인 초기 로딩 속도 최적화(Extreme Performance)** 입니다.

- **목표 성능**: TTFB(Time To First Byte) 20ms 이하 달성.
- **Server Components (RSC) 도입**: `apps/web`, `apps/admin`, `apps/driver` 등 모든 Next.js 15 애플리케이션의 메인 페이지와 핵심 컴포넌트는 Server Component로 설계하여 클라이언트 번들 사이즈와 부하를 최소화합니다.
- **초경량 UI 지향**: 
  - `lucide-react`, `date-fns`와 같은 무거운 외부 라이브러리 사용을 엄격히 금지합니다.
  - 아이콘은 Native API 및 **Unicode Emoji**로 대체하여 로딩 지연을 원천 차단합니다.
- **스타일링**: Tailwind CSS 기반으로 작성하며, Native CSS 활용을 극대화합니다. `packages/ui`에서 공통 디자인 유틸리티를 관리합니다.

## 2. 모노레포 구조 및 상태 관리 (Monorepo & State)

GreenLink는 **Turborepo** 기반으로 구축되었으며 패키지 간 의존성을 엄격히 관리합니다.

- **apps/** : 개별 배포되는 애플리케이션 계층 구역
  - `web` : 소비자용 하이퍼로컬 PWA
  - `admin` : 농가 및 판매자 관리 대시보드
  - `driver` : 신선배송 기사용 전용 앱 (다크 테마 기본)
- **packages/** : 공통 모듈 및 로직 공유 구역 
  - `ui` : 공통 UI 컴포넌트 시스템 (Radix UI + Tailwind)
  - `lib` : API 클라이언트, 전역 상태 관리 로직, 공용 데이터
- **상태 관리 (State Management)**: 전역 상태는 **Zustand**를 사용하며, 필요한 경우 Persistence mode(로컬/세션 스토리지 등)를 적용하여 새로고침 시에도 사용자 컨텍스트를 유지합니다.

## 3. 핵심 비즈니스 로직 (Core Business Logic)

플랫폼의 모든 활동은 '중간 유통을 생략한 하이퍼로컬 산지 직거래'를 완벽히 구현하는 것에 맞춰져 있습니다.

### 3.1. 농장 및 신뢰도 시스템 (Farm & Trust)
- **당근마켓형 직거래 프로필**: 농가는 고유의 프로필을 가지며 소비자 반경 내의 하이퍼로컬 검색에 노출됩니다.
- **신뢰 지표 (온도 시스템)**:
  - **그린 온도 (Green Temp)**: 판매자(농가)의 신뢰도 지표. 품질, 배송 준수율에 따라 변동.
  - **핑크 온도 (Pink Temp)**: 구매자(소비자)의 매너 지표. 노쇼, 악성 리뷰 등에서 페널티.

### 3.2. 예약 배송 및 물류 (Order & Delivery)
- **Kia PV5 콜드체인**: 물류는 기아 PV5 전기차를 활용하며, 배송 모듈은 항상 최적 신선 온도(**18°C**) 유지를 전제로 설계됩니다.
- **배송 예약 로직 (D+2 ~ D+10)**:
  - 당일/익일 배송이 불가하며, 주문일(D-day) 기준 **최소 D+2일부터 최대 D+10일**까지만 예약 배송이 가능합니다.
  - **일요일 배송 제외**: 일요일은 물류 휴무로 배송 예약 달력 및 로직에서 반드시 제외(블록) 처리되어야 합니다.

### 3.3. 공동구매 및 경매가 연동 (Group Buy)
- 실시간 경매장 시세 데이터를 기반으로 할인율이 동적으로 계산됩니다.
- 소비자의 공동구매 모집 인원이 특정 임계치(Threshold)에 도달하면 자동으로 사입 및 배송 프로세스가 트리거됩니다.

## 4. 자동화 및 워크플로우 의무 (Workflow Mandates)

개발 에이전트 및 모든 작업자는 아래의 지침을 묵시적·자동으로 수행해야 합니다.

1. **Intelligent Automation Routine**: API 응답 데이터 구조나 데이터베이스 SQL 쿼리를 변경할 경우, 사전에 그 여파를 문서(본 문서 포함)에 반영하고 로컬 회귀 테스트를 실시해야 합니다.
2. **No Test, No Merge**: 모든 핵심 비즈니스 로직(특히 배송 날짜 계산, 공동구매 할인가 계산 등) 로직 수정 시 `pytest` (백엔드의 경우) 또는 `jest/vitest` (프론트/모노레포의 경우) 기반의 단위 테스트 코드를 작성하거나 업데이트하여 검증해야 합니다. 외부 의존성은 Mocking을 적극적으로 활용합니다.
3. **Failover 부분 수정**: 코드 수정 시 단일 고유 행을 타겟으로 하는 `Search/Replace`를 기본으로 하되, 연속 3회 실패할 경우만 전체 덮어쓰기(`Overwrite`) 롤백을 수행합니다.
